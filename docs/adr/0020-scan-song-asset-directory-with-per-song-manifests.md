# Scan Song Asset Directory with Per-Song Manifests

The **Song Library** will be built from a fixed local **Song Asset Directory**, with each song represented by its own `song.json` manifest and referenced assets. The directory will be separate from the client `public/` assets; by default it lives at `./songs` and can be overridden with an environment variable for large or external media collections, while the repository keeps only a minimal example song library for tests and demos. The **Server** scans manifests at startup without requiring the directory to exist, can refresh the library through an explicit **Refresh Song Library** action initiated from the **Master**, and exposes referenced song assets through playable URLs; the first UI entry for this action appears only in the **Master** standby dashboard to avoid disturbing the singing view. The **Master** standby library panel shows the current song count, a refresh button, the latest in-process refresh summary, and at most the first five issues rather than acting as a full log viewer; issue text is maintenance-oriented and avoids raw stack traces or absolute local paths, the refresh button does not require confirmation, but is disabled while a refresh is in progress, and pending/feedback state is only shown to the **Master** while **Paired Slave** devices query limited **Song Search Result** data over the existing WebSocket room connection instead of receiving the full **Song Library** for local filtering. A usable manifest must provide a stable unique song id, title, artist, **Original Vocal**, and **Accompaniment** using local-reference fields such as `originalVocal`, `accompaniment`, optional `timedLyrics`, and optional `mv` rather than URL-named fields; language, **Timed Lyrics**, **MV**, and extra search text are optional, lyrics are referenced as a separate JSON timed-lyrics asset file rather than embedded in code, and unknown manifest fields are ignored. Manifest asset references must be local files that stay within that song's own directory, keeping songs portable, avoiding remote-asset availability problems, and preventing path escape from exposing unrelated local files; the **Server** validates existence and path boundaries, while real media playability remains a **Master** playback concern reported as **Unplayable Song**. **Song Library Refresh Issue** feedback has blocking and non-blocking levels: blocking issues skip a candidate **Song**, while non-blocking issues such as missing **Timed Lyrics** or **Original Vocal** and **Accompaniment** sharing the same file let the **Song** remain available; missing **MV** is allowed without an issue. Songs are stored in a stable order using optional manifest `sortTitle`, then `title`, `artist`, and `id`, rather than relying on filesystem traversal order, and **Song Search** matches title, artist, and optional manifest search text with simple case-insensitive whitespace-normalized containment over precomputed normalized text, rather than hidden fuzzy matching, generated-pinyin matching, a database, or a complex search index. Empty and non-empty searches are requested after a short input debounce with no minimum query length and include a request id so **Slave** can ignore stale out-of-order responses; empty searches return the first 50 songs in stable order, while non-empty searches rank title prefix matches before title contains, artist contains, and search-text contains, with each group using stable order. Search responses return at most 50 song summaries without playback asset URLs, lyrics, queue-derived counts, or match-highlight metadata, include whether more matches exist, and do not provide pagination in the first implementation, so **Slave** devices can prompt for a narrower query without receiving a large library snapshot. Adding a song to the queue still sends only the selected song id, and the **Server** builds the **Queued Song** snapshot from the current **Song Library** rather than trusting search result data from the **Slave**; if a search result has become stale because the song left the current **Song Library**, the add request is rejected instead of creating a queue item. A successful refresh advances a lightweight song-library version so **Slave** devices can re-run their current search without the **Server** storing per-device query state. A refresh may run while a **Current Song** is playing because it only affects future song selection and does not rewrite existing **Queued Song** or **Current Song** snapshots; invalid candidate songs are skipped with **Song Library Refresh Issue** feedback visible on the **Master**, **Slave** devices show a plain empty state when search has no available source, show a plain no-matches state when the current query matches no songs, keep the current search text after adding a song, use a short per-song pending state to prevent accidental double-submit while preserving intentional duplicate requests, rely on queue updates rather than success toasts for add-song feedback, and may derive a non-persistent "in queue" count from the current in-process **Playback Queue** that excludes the **Current Song** and does not prevent duplicate song requests; startup may produce an empty **Song Library** when there is no usable source, and a later refresh that finds no usable songs is treated as a failed refresh that keeps the previous **Song Library**.
